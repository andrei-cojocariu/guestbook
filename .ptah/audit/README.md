# Ptah Audit Knowledge Base

This directory is the decentralized, verified map of the `guestbook` codebase.
It is generated by the Forensic Auditor Worker (pipeline stage 1, `Analyze`) and
is the single source of truth downstream implementation and review workers are
gated against. It is committed to the main-task branch so every worker inherits
the same map.

## Layout

```text
.ptah/audit/
  system.md              Stack, architecture, boundaries, detection evidence
  standards.md           Enforceable standard matrix (transitional mode)
  legacy_debt.md         Risk-first debt ledger + Strangler Fig seams
  INDEX.md               Feature-to-file map, kept current every run
  features/<slug>.md     One capability, BDD format, bidirectional links
  files/<path>.md        One meaningful source file, mirroring its path
```

## Conventions

- One H1 per file, fenced code carries a language, tables are aligned.
- `features/*.md` follow the BDD standard: link frontmatter, User Story,
  Gherkin scenarios. Prose behavior descriptions are non-compliant.
- `files/*.md` name their `used_by` / `part_of` / `touches` links and a blast
  radius. Keep each file short; link, do not inline source.
- Read-only w.r.t. product code. Nothing outside `.ptah/audit/` is written here.

## Scope of this run

Scope: `full` (whole project). This refresh reconciled the KB against HEAD
(`76301fe`); product code is unchanged since the prior audit, so existing entries
were verified rather than rewritten. Gaps filled this run: `autoload.php` global
coupling (database + url helper) and the absence of any CI pipeline. The effective
product surface remains the guestbook code (form, timeline, persistence, CI
form-validation/sanitization). Framework `system/` and `user_guide/` are treated
as vendor/legacy and are not documented file-by-file.
