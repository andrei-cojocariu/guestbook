## Findings

1. `index.php:56` — ENVIRONMENT is resolved from $_SERVER['CI_ENV'], which Apache/mod_php does NOT populate from the container's OS/Docker CI_ENV variable. I reproduced live: with `docker compose run -e CI_ENV=production ... app` and unreachable DB creds, a real HTTP GET of /index.php/guestbook returns 'A Database Error Occurred', 'Unable to connect to your database server', and 'mysqli' to the client. So ENVIRONMENT falls back to 'development', db_debug at database.php:109 resolves TRUE, and SQL/connection detail leaks under the actual deploy SAPI. The database.php-only change cannot satisfy the criterion because the environment gate lives in the front controller. The branch's own index.php.md and EnvSecretManagementTest.php.md document this same failure.
   - rule: BDD contract features/secret-management.md Scenario 3 ("Given CI_ENV is set to production ... Then db_debug is FALSE And no SQL error detail is returned to the client"); TAC bullet 2; legacy_debt.md SEC-5 #db-debug-leak
   - fix: Make ENVIRONMENT observe the deploy SAPI: resolve CI_ENV via getenv() (the branch already confirms getenv() DOES propagate under Apache/mod_php) with the $_SERVER lookup as fallback, e.g. define('ENVIRONMENT', getenv('CI_ENV') ?: (isset($_SERVER['CI_ENV']) ? $_SERVER['CI_ENV'] : 'development')); OR pass CI_ENV through at the web-server layer (Apache SetEnv/PassEnv in the Dockerfile/image). index.php is inside the feature's declared implemented_by surface, so editing it is in scope. Then re-run the gate with MYSQL_ROOT_PASSWORD set and confirm the CI_ENV=production HTTP request no longer renders any DB/SQL error text, while CI_ENV unset still does (the contrast check).

3. `docker-compose.yml:66` — The `app` service forwards neither CI_ENV nor DB_HOSTNAME/DB_USERNAME/DB_PASSWORD/DB_DATABASE into the container. As the branch's own database.php.md 'Known gap' note concedes, a live `docker compose up` therefore falls through to the non-secret local defaults (empty password) and never exercises the env-var contract against the real db service. The env contract is thus unwired in the actual running topology, and CI_ENV=production is never set in the deploy path — leaving the SEC-5 fix inoperative even independent of the SAPI bug above. The developer deferred this as 'infra scope out of this task's file list,' but Execution Plan step 3 assigns it to this task.
   - rule: Execution Plan step 3 ("Force db_debug=FALSE outside development and ensure CI_ENV=production is set in the deploy path; document the required env vars in the container"); TAC bullet 3 (env-var contract resolves inside the frozen runtime)
   - fix: Add an `environment:` block to the `app` service forwarding CI_ENV plus DB_HOSTNAME/DB_USERNAME/DB_PASSWORD/DB_DATABASE, sourced from the git-ignored .env (same pattern as the db service's MYSQL_ROOT_PASSWORD), so `docker compose up` boots the app against its DB using env-sourced credentials with CI_ENV=production in the deploy path. Document the required vars (e.g. in .env.example).
